CUB3D PROJECT NOTES
===================

1. OVERVIEW
-----------
This Cub3D project is a Wolfenstein-style 3D raycasting engine built in C with the MiniLibX library. It reads a `.cub` configuration map, validates it, loads textures, and renders a first-person view by casting one ray per screen column.

The engine includes:
- map parsing and validation
- texture loading for walls, floor, and ceiling
- DDA-based raycasting for walls
- smooth WASD movement, strafing, and rotation
- mouse-look with a controllable capture/release feature
- collision detection that stops the player before the wall
- floor/ceiling rendering including optional texture support

Key files:
- `src/main.c` - entry point
- `inc/cub3d.h` - shared structures and function declarations
- `src/parsing/*` - map reading and validation
- `src/parsing/header_parsing_helper.c` - header dispatch and directive parsing
- `src/init/*` - MLX and window setup
- `src/movement/*` - input and player movement
- `src/raycasting/*` - DDA raycasting logic
- `src/render/*` - drawing frame buffer and walls
- `src/texture/texture_load.c` - texture loading

---

2. ENTRY POINT & PROGRAM FLOW
----------------------------
File: `src/main.c`

`main()` does the following:
1. checks command-line count: `./cub3D <map.cub>`
2. zeroes the `t_data` structure using `ft_bzero`
3. runs `verifications()` to validate the file, parse the map, initialize MLX, and create the image buffer
4. loads textures with `load_all_textures()`
5. creates the window with `setup_window()`
6. sets `mlx_loop_hook` to `render_loop`
7. enters `mlx_loop`

If initialization fails at any stage, `destroy_game()` is used to clean memory and exit safely.

---

3. DATA STRUCTURES
------------------
File: `inc/cub3d.h`

`t_map` holds map and player data:
- texture paths: `no_path`, `so_path`, `we_path`, `ea_path`
- optional floor/ceiling texture paths: `floor_tex_path`, `ceil_tex_path`
- `floor_rgb`, `ceil_rgb`
- `base_dir` for map-relative path calculation
- `grid` array of map rows
- `height`, `width`
- player start position and direction: `player_x`, `player_y`, `dir_x`, `dir_y`, `plane_x`, `plane_y`
- `player_dir` char for spawn orientation

`t_data` holds game state and MLX handles:
- `mlx_ptr`, `win_ptr`
- `img` buffer for screen drawing
- `map` data
- textures: `tex[4]` for walls, `floor_tex`, `ceil_tex`
- key states for movement and rotation
- mouse capture state: `mouse_captured`, `mouse_initialized`
- `pitch_offset` for vertical look

`t_dda` stores per-ray DDA state:
- ray origin and direction
- current map cell indices
- `delta_dist_x`, `delta_dist_y`
- step direction and side distances
- `side` to know whether hit was vertical or horizontal
- `perp_wall_dist`

4. MAP PARSING & VALIDATION
---------------------------
Files: `src/parsing/file_reader.c`, `src/parsing/header_parsing.c`, `src/parsing/map_extraction.c`, `src/parsing/map_validation.c`, `src/parsing/map_padding.c`, `src/parsing/player_parsing.c`

4.1 `.cub` file validation
- `validate_file()` in `src/parsing/file_validation.c` ensures the file ends with `.cub` and can be opened.

4.2 Reading all lines
- `read_map_file()` reads the file into a dynamic array of strings using `get_next_line`.
- It expands buffer capacity dynamically and returns `lines` and `count`.

4.3 Header parsing
- Header tokens are parsed by `parse_header()` in `src/parsing/header_parsing.c`, with line dispatch performed by `parse_header_line()` in `src/parsing/header_parsing_helper.c`.
- Supported header keys:
  - `NO`, `SO`, `WE`, `EA`: wall texture paths
  - `F`: either a floor color in `R,G,B` or a floor texture path
  - `C`: either a ceiling color in `R,G,B` or a ceiling texture path
- Legacy keys `FT` and `CT` are also accepted for floor/ceiling textures.
- Duplicate or unknown header keys now produce a clear `Error: invalid or duplicate header key` failure.
- Each parser stores values in `t_map`.

4.4 Map extraction
- `collect_map_lines()` builds the raw map grid from the remaining lines.
- `strip_line()` removes newline characters.
- `find_map_bounds()` locates the first and last non-empty map lines.

4.5 Map validation
- `validate_map_chars()` ensures every cell is valid: `0`, `1`, ` `, `N`, `S`, `E`, or `W`.
- `is_map_line()` detects map content during header processing.
- `check_map_enclosed()` verifies the map is closed using flood-fill.
  - It copies the grid and recursively fills reachable tiles.
  - If it reaches a `' '` or invalid boundary, the map is declared not enclosed.

4.6 Map padding
- `pad_map_grid()` wraps the entire map with a border of `1` tiles.
- This is important because it simplifies boundary checks and ensures DDA rays never escape into undefined space.
- The padding function also adds a `1` column on each side and a full `1` row at top and bottom.

4.7 Player position extraction
- `extract_player_data()` searches for exactly one `N`, `S`, `E`, or `W`.
- `set_player_at()` stores the player position at the center of the map cell: `col + 0.5`, `row + 0.5`.
- The spawn marker is replaced with `'0'` after use.
- `set_player_dir()` initializes the direction vector and camera plane based on orientation:
  - `N`: `dir_y = -1.0`
  - `S`: `dir_y = +1.0`
  - `E`: `dir_x = +1.0`
  - `W`: `dir_x = -1.0`
- The camera plane is set to `0.66` times the perpendicular direction vector.

5. INITIALIZATION & CLEANUP
--------------------------
Files: `src/init/init.c`, `src/init/cleanup.c`, `src/movement/key_handler.c`, `src/movement/mouse_capture.c`

5.1 MLX setup
- `init_mlx()` calls `mlx_init()` and stores the handle in `data->mlx_ptr`.
- `init_image()` creates the render buffer with `mlx_new_image()` and reads its data address.
- `setup_window()` creates the window and installs hooks.

5.2 Hook setup
- `setup_hooks()` attaches keyboard, mouse movement, and close callbacks:
  - key press event: `handle_keypress`
  - key release event: `handle_keyrelease`
  - mouse move event: `handle_mouse_move`
  - mouse leave event: `handle_mouse_leave`
  - window close event: `close_window`
- It also calls `mouse_capture()` to hide the cursor and lock it to the window center.

5.3 Cleanup
- `destroy_images()` destroys the MLX image buffer and all loaded textures.
- `destroy_game()` destroys the window and MLX display, frees the MLX pointer, and calls `free_map()`.
- `free_map()` frees map strings, the grid, and resets the map struct.

---

6. TEXTURE LOADING
------------------
File: `src/texture/texture_load.c`

6.1 Loading wall textures
- `load_wall_texture()` tries to load each wall texture path into `data->tex[0..3]`.
- It uses `load_one_texture()` and `mlx_xpm_file_to_image()`.
- If a wall texture fails to load, the program prints `Error: failed to load wall textures` and initialization fails.

6.2 Loading floor/ceiling textures
- If floor or ceiling texture paths are provided via `F`, `C`, `FT`, or `CT`, `load_all_textures()` loads `floor_tex` and `ceil_tex`.
- If a floor texture path fails to load, the program prints `Error: failed to load floor textures` and exits.
- If a ceiling texture path fails to load, the program prints `Error: failed to load ceiling textures` and exits.
- If no floor/ceiling textures are provided, solid colors are used instead.

6.3 Full path building
- `make_full_path()` builds an absolute path from either the current working directory or a relative path.
- This helps MLX find `.xpm` files reliably.

7. INPUT & MOVEMENT
-------------------
Files: `src/movement/key_handler.c`, `src/movement/movement.c`, `src/movement/mouse_capture.c`, `src/movement/key_handler_helper.c`

7.1 Keyboard state tracking
- The game tracks keys using boolean fields in `t_data`:
  - `key_w`, `key_a`, `key_s`, `key_d`
  - `key_left`, `key_right`, `key_up`, `key_down`
  - `key_ctrl`
- `handle_keypress()` sets the state to pressed and releases the mouse when Ctrl is pressed.
- `handle_keyrelease()` clears the key state and recaptures the mouse when Ctrl is released.

7.2 Continuous input processing
- `process_input()` is called every frame in `render_loop()`.
- Movement actions are applied while the associated key is held:
  - `W`/`S`: move forward/backward
  - `A`/`D`: strafe left/right
  - left/right arrows: rotate view
  - up/down arrows: pitch view vertically

7.3 Movement math
- `move_player()` moves along `map->dir_x` and `map->dir_y`.
- `strafe_player()` moves along `map->plane_x` and `map->plane_y`.
- Movement is multiplied by `MOVE_SPEED` and `direction` (`+1` or `-1`).

7.4 Collision handling
- `can_move_to()` checks the four corner positions around the proposed destination using `COLLISION_BUFFER`.
- This keeps the player from approaching too close to walls.
- It converts the coordinates to integer map cells and returns `0` if any cell is a wall.
- This is the key feature that stops movement just before the wall.

7.5 Rotation
- `rotate_player()` rotates both the direction vector and the camera plane.
- The rotation uses a 2D rotation matrix with `cos(angle)` and `sin(angle)`.
- This makes the view rotate smoothly while preserving the camera plane.

7.6 Mouse capture and release
- `mouse_capture()` hides the cursor and moves it to the center of the window.
- `mouse_release()` shows the cursor again.
- `handle_keypress()` releases the mouse when `LCTRL_KEY` is pressed.
- `handle_keyrelease()` recaptures the mouse when Ctrl is released.
- This is the exact feature you asked for: Ctrl gives back the mouse.

7.7 Mouse look implementation
- `handle_mouse_move()` ignores movement unless the mouse is captured.
- It clamps the raw mouse position to the window size.
- The delta from the center is computed and passed to `handle_mouse_delta()`.
- `handle_mouse_delta()` rotates the player horizontally and adjusts `pitch_offset` vertically.
- After processing movement, it re-centers the mouse pointer.

8. RAYCASTING & WALL RENDERING
------------------------------
Files: `src/raycasting/dda.c`, `src/raycasting/dda_helper.c`, `src/raycasting/raycasting.c`, `src/render/render_draw.c`, `src/render/render_draw_helper.c`

8.1 Screen-space raycasting
- `render_raycasting()` loops one ray for every screen column from `x = 0` to `x < WIN_WIDTH`.
- For each column it:
  1. initializes a `t_dda` ray with `init_dda_for_ray()`
  2. advances the ray through the grid with `perform_dda()`
  3. draws the resulting wall slice with `draw_wall_column()`

8.2 Ray direction calculation
- `compute_ray_direction()` computes the ray vector using the camera plane:
  - `camera_x = 2.0 * x / WIN_WIDTH - 1.0`
  - `ray_dir = dir + plane * camera_x`
- This formula gives a different ray direction for every column.

8.3 DDA initialization
- `init_dda_for_ray()` sets the ray origin at the player and the DDA map cell.
- It computes step distances with `init_dda_distances()`:
  - `delta_dist_x = fabs(1/ray_dir_x)` or a large infinity value if ray is vertical
  - `delta_dist_y = fabs(1/ray_dir_y)` or infinity if ray is horizontal
- It computes the first side distances `side_dist_x` and `side_dist_y`.
- `step_x` and `step_y` are either `+1` or `-1` depending on the ray direction.

8.4 DDA walking loop
- `perform_dda()` chooses whether to step in X or Y each iteration.
- It compares `side_dist_x` and `side_dist_y` and moves along the smaller one.
- When the ray enters a wall cell (`is_wall_cell()` returns true), the loop stops.
- The DDA side value `dda->side` indicates which axis was crossed.

8.5 Perpendicular wall distance
- `compute_perp_wall_dist()` computes the distance from the player to the wall plane:
  - if `side == 0`: `side_dist_x - delta_dist_x`
  - else: `side_dist_y - delta_dist_y`
- This is used to calculate the correct wall slice height and avoid distortion.
- A large `infinity` distance is returned when needed by `get_dda_infinity()`.

8.6 Wall slice rendering
- `draw_wall_column()` computes the line height using `WIN_HEIGHT / perp_wall_dist`.
- It clamps start and end of the line to screen bounds.
- If no wall textures are loaded, it draws a solid gray wall with darker shading for `side == 1`.
- If textures are available, it calls `render_draw_textured_column()`.

8.7 Texture selection and sampling
- `get_wall_tex_idx()` selects wall texture index based on hit side and ray direction:
  - X-side hits choose `WE` vs `EA`
  - Y-side hits choose `NO` vs `SO`
- `calc_wall_x()` computes the exact position along the wall where the ray hit.
- `calc_tex_x()` maps that position to a texture column, flipping the texture horizontally depending on direction.
- The vertical texture loop samples pixels from the texture and writes them to the screen.

9. FLOOR & CEILING RENDERING
---------------------------
Files: `src/render/render_floor_ceiling.c`, `src/render/render_floor_ceiling_helper.c`

9.1 Solid floor and ceiling
- `render_floor_and_ceiling()` fills the top half of the screen with ceiling color and the bottom half with floor color.
- `pitch_offset` changes the center line to simulate vertical looking.

9.2 Textured floor and ceiling
- If `floor_tex` or `ceil_tex` are loaded, the engine uses `render_floor_ceiling_textured()`.
- It performs a simple projection for each screen row:
  - compute `row_dist` based on distance from vertical center
  - compute texture sampling coordinates from the player position and ray directions
  - sample `floor_tex` for bottom rows and `ceil_tex` for top rows
- This is a cheaper alternative to full 3D floor casting, but it still produces a textured floor/ceiling effect.

10. IMPORTANT IMPLEMENTATION POINTS
----------------------------------

10.1 Raycasting management
- The engine uses a classic raycasting pipeline: one ray per vertical column.
- DDA walks across the grid one tile boundary at a time until a wall cell is hit.
- The ray direction is computed from the player direction and camera plane.
- The camera plane is perpendicular to the direction vector and controls the field of view.

10.2 Stopping before the wall
- Collision detection does not allow the player to move into a wall.
- `can_move_to()` tests positions offset by `COLLISION_BUFFER` in both axes.
- It checks adjacent cells at all four corners around the intended location.
- This is what makes the player stop before the wall and prevents clipping.

10.3 Mouse capture / Ctrl release
- Mouse capture is managed explicitly with `mouse_capture()` and `mouse_release()`.
- When the game starts, the mouse is hidden and recentered.
- Pressing left Ctrl releases the mouse and makes it visible again.
- Releasing Ctrl calls `mouse_capture()` again to restore control.
- The code uses `mouse_initialized` so the first motion event recenters the pointer without causing a jump.

10.4 Design choices worth mentioning
- `pad_map_grid()` adds a border of walls around the original map to simplify flood-fill and raycasting.
- `make_inner_row()` safely pads short map lines with spaces so padded rows preserve the original layout without memory errors.
- `player_x` and `player_y` are centered inside the map cell: this is why movement and ray origin are stable.
- The engine supports optional floor/ceiling textures in addition to solid colors.
- If wall textures fail to load, initialization now fails with a clear message instead of silently falling back.
- Floor and ceiling texture loading also fail fast with separate error messages for floor vs ceiling.

11. HOW TO TALK ABOUT IT IN PRESENTATION
--------------------------------------
Use this structure:
1. Explain the goal: create a 3D view from a 2D map using raycasting.
2. Describe input: `.cub` file contains textures, colors, and the map layout.
3. Describe parsing and validation:
   - headers first, then map lines
   - check valid characters and ensure the map is enclosed
   - locate the player spawn, save orientation, and pad the map.
4. Describe rendering:
   - each screen column is a ray
   - DDA steps through grid cells until it finds a wall
   - wall slice height is based on perpendicular distance
   - textures are sampled and drawn to the screen
5. Describe movement:
   - WASD + arrows for movement and rotation
   - collision buffer prevents walking into walls
   - mouse look rotates the view and adjusts vertical pitch
   - Ctrl releases the mouse pointer safely
6. Mention robustness:
   - memory cleanup in `destroy_game()`
   - invalid texture paths stop initialization with explicit error messages
   - safe `.cub` validation and map enclosure checks

13. QUICK PRESENTATION NOTES
---------------------------
- The raycasting engine is built on `DDA` and uses `MINILIBX` for graphics.
- The `.cub` file defines textures and map layout.
- Map validation includes both character checks and enclosure checks.
- Player movement uses a collision buffer to stop before walls.
- Mouse capture is intentionally released with Ctrl and recaptured on release.
- Texture loading is flexible: walls may be textured, and floors/ceilings can also use textures.
- Invalid wall or floor/ceiling texture paths now stop initialization with explicit errors instead of silently falling back.
- The renderer draws a full image buffer and pushes it in one call to the window.
