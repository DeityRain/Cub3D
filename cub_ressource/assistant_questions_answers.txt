Questions and Correct Answers
=============================

1) Question: What stops the player from moving into a wall?
   Answer: The movement code computes a candidate new position and calls `can_move_to()` before updating `player_x` or `player_y`. `can_move_to()` checks nearby map cells and rejects moves that would enter a wall.

2) Question: In this project, when the player tries to move, what function decides whether the new position is allowed or blocked by a wall?
   Answer: `can_move_to()` in `src/movement/movement.c` decides whether the planned `x,y` position is valid before updating the player coordinates.

3) Question: When `move_player()` tries to move the player, why does it check X and Y separately instead of updating both at once?
   Answer: Because each axis can be blocked independently by walls. Checking separately allows the player to slide along a wall instead of being blocked completely when only one direction is invalid.

4) Question: What is the purpose of `COLLISION_BUFFER` in the movement collision check?
   Answer: `COLLISION_BUFFER` is a safety margin around the player. It prevents the player from getting too close to a wall cell, avoiding clipping and keeping the player a small distance away from the wall.

5) Question: In `move_player()` and `strafe_player()`, what is the difference between `dir_x/dir_y` and `plane_x/plane_y`?
   Answer: `dir_x/dir_y` is the player’s forward/backward facing direction. `plane_x/plane_y` is a perpendicular sideways vector used for strafing left/right.

6) Question: In `rotate_player()`, why do we save `old_dir_x` and `old_plane_x` before updating the direction and plane vectors?
   Answer: Because the new `dir_y` and `plane_y` formulas need the original `dir_x` and `plane_x` values. Saving them prevents losing the old components when the code overwrites `dir_x` and `plane_x`.

7) Question: In `can_move_to()`, why is the candidate position checked with both `x + COLLISION_BUFFER` and `x - COLLISION_BUFFER` (and the same for `y`)?
   Answer: To check the four edges around the candidate position. This ensures no wall lies within the buffer zone in any direction around the player.

8) Question: In `read_map_file()` from `file_reader.c`, why does the code call `expand_lines_buffer()` one more time after reading all lines, before setting `arr[n] = NULL`?
   Answer: Because `arr[n] = NULL` needs one extra slot in the array. If capacity equals `n`, there is no free space for the terminator, so the buffer is expanded first.

9) Question: In `read_map_file()`, why is `arr[n] = NULL` useful after reading the file lines?
   Answer: It terminates the array of strings, just like `\0` terminates a C string. It marks the end of the list so code can stop when it reaches `NULL`.

10) Question: In `read_and_buffer_lines()`, what happens when `get_next_line(fd)` returns `NULL`?
    Answer: The loop breaks and the function returns `1`, meaning end of file has been reached and reading finished normally.

11) Question: In `perform_dda()`, why does the loop compare `side_dist_x` and `side_dist_y` to choose whether to step in X or Y first?
    Answer: Because the smaller side distance identifies which grid boundary the ray hits first. The algorithm steps to the nearest next cell along the ray.

12) Question: In `init_dda_distances()`, why is `delta_dist_x` computed as `fabs(1.0 / ray_dir_x)` and `delta_dist_y` as `fabs(1.0 / ray_dir_y)`?
    Answer: Because these are the absolute positive distances needed to travel one whole cell in X or Y along the ray. `fabs()` ensures the step size is positive regardless of ray direction.

13) Question: What is the simplest way to explain the concept of DDA?
    Answer: DDA follows a ray through the grid one cell boundary at a time, always stepping to the nearest next wall boundary, until it finds a wall.

14) Question: Why do we use the hypotenuse for the wall distance?
    Answer: Because the ray usually hits the wall at an angle. The true distance is the straight-line length between player and wall, which is the hypotenuse of the right triangle formed by X and Y differences.

15) Question: Why does the code set `dda->map_x = (int)dda->pos_x` and `dda->map_y = (int)dda->pos_y`?
    Answer: Because the map is a grid of cells, and truncating the player’s floating position gives the current map cell indices. This is the starting cell for the ray.

16) Question: What does the `side` variable mean in DDA, and why is it important?
    Answer: `side` indicates whether the ray hit a vertical grid line (`0` for X side) or a horizontal grid line (`1` for Y side). It is important for calculating the correct perpendicular distance and for choosing the correct wall shading or texture orientation.

17) Question: Why do `init_dda_x()` and `init_dda_y()` compute `side_dist_x` and `side_dist_y` using the distance to the first grid line in the ray direction?
    Answer: Because the ray starts inside a cell. The first step must travel from the player position to the next vertical or horizontal wall boundary. These initial distances tell DDA how far to go before the first grid boundary hit.
